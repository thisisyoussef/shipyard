# Software Factory Skills

A bootleg implementation of the Athena Software Factory pattern вҖ” 10 skills that chain
together to go from feature spec вҶ’ tested PR-ready code, following your existing codebase
conventions.

## Skills

| # | Skill | What It Does | Typical Trigger |
|---|-------|-------------|----------------|
| 1 | `repo-constitution` | Index repo conventions into a constitution | "learn this repo" |
| 2 | `feature-model` | Enrich a spec with codebase context | "parse this spec" |
| 3 | `execution-planner` | Create ordered execution plan | "plan this feature" |
| 4 | `migration-gen` | Generate safe DB migrations + schema contract | "generate migration" |
| 5 | `api-gen` | Generate backend code + API contract | "generate API" |
| 6 | `config-gen` | Generate flags, permissions, env vars | "set up config" |
| 7 | `frontend-gen` | Generate UI components from API contract | "generate frontend" |
| 8 | `test-suite-gen` | Generate multi-layer test suite | "generate tests" |
| 9 | `pr-assembler` | Verify + assemble PR + QA checklist | "assemble PR" |
| 10 | `factory-orchestrator` | Run full pipeline end-to-end | "run the factory" |

## Quick Start

### 1. Index your repo (one-time)

```
> Index this repo's conventions
```

This creates `.factory/CONSTITUTION.md` with your codebase's conventions.

### 2. Provide a feature spec

```json
{
  "entity": "PaymentMethod",
  "fields": [
    {"name": "type", "kind": "enum", "values": ["credit_card", "ach", "wire"]},
    {"name": "label", "kind": "string"},
    {"name": "is_default", "kind": "boolean"},
    {"name": "vault_ref", "kind": "string", "sensitive": true}
  ],
  "permissions": "payment_methods:manage",
  "audit": true,
  "integrations": ["vault"],
  "ui": {"list": true, "detail": true, "form": true}
}
```

### 3. Run the factory

```
> Run the factory on this spec
```

Or run stages individually:
```
> Create a feature model from this spec
> Plan the execution
> Generate the migration
> Generate the API
> Generate the frontend
> Generate the tests
> Assemble the PR
```

## Key Concepts

### Typed Contracts Between Stages

Stages don't share context вҖ” they communicate through JSON contracts:
- **Schema Contract** (`schema-contract.json`): migration вҶ’ api-gen
- **API Contract** (`api-contract.json`): api-gen вҶ’ frontend-gen, test-suite-gen
- **Feature Model** (`feature-model.json`): consumed by all stages

### Exemplar-Driven Generation

Code is generated by copying the structure of existing code in your repo (the "exemplar")
and adapting it for the new entity. This produces code that matches your team's style
because it was structurally derived from your team's code.

### Constitution Over Convention

The constitution documents what IS in your codebase, not what should be. It's extracted
by analyzing your actual code, not by following generic best practices.

### Verify Continuously

Each stage verifies its own output using your project's real toolchain (your eslint config,
your tsconfig, your test runner). Not generic checks вҖ” your checks.

## Adapting for This Workspace

This project uses:
- **Language**: TypeScript
- **Runtime**: Local Node CLI under `shipyard/`
- **Testing**: Vitest
- **Build**: TypeScript compiler
- **Current surface**: typed tools, context discovery, checkpoints, and tracing

The constitution should capture this workspace's specific patterns:
- nested app layout under `shipyard/`
- persistent CLI loop and typed tool registry
- local trace and checkpoint state under `shipyard/.shipyard/`
- root-level helper harness in `.ai/`

## Limitations vs. Full Athena Factory

This bootleg version trades infrastructure for simplicity:

| Athena Factory | Bootleg Version |
|---------------|----------------|
| Dedicated orchestration server | Skills in your AI coding tool |
| Qdrant vector store for exemplars | Markdown files + grep |
| Tree-sitter AST analysis | LLM reading source files |
| Parallel stage execution | Sequential (or manual parallel) |
| Automated feedback loops | Manual exemplar score updates |
| Session manager (Epoch-Runner) | Fresh context per skill invocation |
| 3-tier model router | Whatever model your tool uses |

The architecture is the same. The implementation is simpler. The key insight is preserved:
**conventions as auto-triggered context, typed contracts between stages, verify at every step.**
