# Migration Generator — Safe Schema Changes + Schema Contract

## Purpose

Generate database migrations that are purely additive and zero-downtime safe, following
expand-and-contract methodology. Produce a typed Schema Contract that downstream skills
(api-gen, test-suite-gen) consume to ensure type alignment across layers.

## Prerequisites

Load from `.factory/current/`:
- `feature-model.json` — field definitions, types, constraints

Load from `.factory/CONSTITUTION.md` — migration conventions, ORM, naming

## Core Principles

### Expand-and-Contract (Non-Negotiable)

Every migration must be **purely additive** in the expand phase:
- Add columns, tables, indexes — never drop or rename in the same migration
- New columns must be NULLABLE or have DEFAULT values
- Contracting (dropping old columns) happens in a separate, later migration

Why: This allows the old application code to keep running during deployment. The new code
writes to both old and new columns, then a cleanup migration removes the old.

### Database Constraints Are Law

Enforce data integrity at the database level, not just the application:
- `NOT NULL` — if a field is required, enforce it in the schema
- `CHECK` constraints — for enums, ranges, format validation
- `UNIQUE` constraints — if business logic requires uniqueness
- `FOREIGN KEY` — with appropriate ON DELETE behavior
- `DEFAULT` — for fields with sensible defaults

Application-level validation is *advice* that can be bypassed. DB constraints are *law*.

### Index Strategy

- Create indexes `CONCURRENTLY` (PostgreSQL) to avoid table locks
- Index foreign keys (they're used in joins — always)
- Index columns used in WHERE clauses and ORDER BY
- Composite indexes: leftmost column should be the most selective

## Generation Process

### Step 1: Read Feature Model

Extract from `feature-model.json`:
- `database.table` — table name
- `database.columns` — column definitions with types and constraints
- `database.indexes` — required indexes

### Step 2: Determine Migration Format

From the constitution, identify:
- **Raw SQL**: produce `.sql` files
- **Prisma**: produce `schema.prisma` additions
- **TypeORM**: produce migration class
- **Knex**: produce `exports.up` / `exports.down`
- **Drizzle**: produce schema definition

### Step 3: Generate Migration

Follow this template structure (adjust for ORM):

```sql
-- Migration: create_[table_name]
-- Generated by factory pipeline
-- Strategy: expand-and-contract (expand phase)

BEGIN;

CREATE TABLE [table_name] (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  -- fields from feature model
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes (CONCURRENTLY for zero-downtime)
CREATE INDEX CONCURRENTLY idx_[table]_[column]
  ON [table_name]([column]);

COMMIT;
```

### Step 4: Generate Schema Contract

The Schema Contract is a typed artifact consumed by api-gen and test-suite-gen.
Save as `.factory/current/schema-contract.json`:

```json
{
  "table": "payment_methods",
  "columns": [
    {
      "name": "id",
      "db_type": "UUID",
      "typescript_type": "string",
      "nullable": false,
      "has_default": true,
      "in_create_dto": false,
      "in_update_dto": false,
      "in_response_dto": true
    },
    {
      "name": "type",
      "db_type": "VARCHAR(20)",
      "typescript_type": "'credit_card' | 'ach' | 'wire'",
      "nullable": false,
      "has_default": false,
      "check": "type IN ('credit_card','ach','wire')",
      "in_create_dto": true,
      "in_update_dto": true,
      "in_response_dto": true
    },
    {
      "name": "vault_ref",
      "db_type": "VARCHAR(255)",
      "typescript_type": "string",
      "nullable": false,
      "sensitive": true,
      "in_create_dto": true,
      "in_update_dto": false,
      "in_response_dto": false
    }
  ],
  "indexes": [
    { "name": "idx_payment_methods_merchant", "columns": ["merchant_id"], "concurrent": true }
  ],
  "foreign_keys": [
    { "column": "merchant_id", "references": "merchants(id)", "on_delete": "CASCADE" }
  ]
}
```

Key rules for the Schema Contract:
- **Sensitive fields** (`sensitive: true`) must be excluded from response DTOs
- **Fields with defaults** are optional in create DTOs
- **id, created_at, updated_at** are never in create/update DTOs

### Step 5: Verify

Run these checks (deterministic — no LLM):

```bash
# SQL syntax validation
pg_format --check migration.sql 2>&1 || echo "SYNTAX ERROR"

# Safety checks (if squawk is available)
squawk migration.sql

# Manual checklist if tools aren't available:
# ✅ No DROP COLUMN or DROP TABLE
# ✅ No ALTER COLUMN ... SET NOT NULL (without DEFAULT)
# ✅ Indexes use CONCURRENTLY
# ✅ Foreign keys reference existing tables
# ✅ Column types match feature model exactly
```

Present verification results to user.

## Output Files

Save to `.factory/current/`:
```
.factory/current/
├── migrations/
│   └── [timestamp]_create_[table].sql  (or ORM-specific format)
├── schema-contract.json
└── migration-verification.md  (check results)
```

## Safety Warnings

Always flag these to the human:
- Any migration touching an existing table (ALTER TABLE)
- Any migration on a table with >1M rows (index creation time)
- Any migration involving enum changes (type-specific concerns)
- Any foreign key referencing a table not in the current schema search path
